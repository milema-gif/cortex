**Design decision (IMPORTANT):** The bookmark DOES advance past failed observations because they are now tracked in sync_failures. If the bookmark didn't advance, a permanently-failing observation would block all subsequent observations forever. The invariant is: every observation_id <= bookmark has EITHER been successfully embedded OR is tracked in sync_failures. The retry phase processes sync_failures entries before new observations each cycle.

**Rewrite src/core/sync.ts syncNewObservations:**

1. At the TOP of the function (before querying new observations), add a retry phase:
   - Query `SELECT * FROM sync_failures WHERE status = 'pending' ORDER BY observation_id ASC`
   - For each pending failure:
     - Call /embed. Check response.ok.
     - If response.ok: DELETE the sync_failures row. Log success.
     - If !response.ok OR fetch throws: increment attempt_count, update last_error (use response.statusText or error message), update last_attempt_at. If attempt_count >= config.syncMaxRetries, set status='parked' and log warning.

2. Fix the main observation loop:
   - Capture the response: `const response = await fetch(...)`
   - Check `response.ok`:
     - If true: advance bookmark (existing behavior)
     - If false: INSERT into sync_failures (observation_id, last_error=`HTTP ${response.status}: ${await response.text()}`, attempt_count=1, status='pending'). Still advance bookmark (observation is tracked).
   - In the catch block (network error): INSERT into sync_failures (observation_id, last_error=error.message, attempt_count=1, status='pending'). Still advance bookmark.

3. Use prepared statements for sync_failures operations (INSERT OR REPLACE, DELETE, UPDATE) -- create them once before the loops.

**Fix embeddingBackfill similarly:**
   - Capture response, check response.ok
   - Only increment successCount when response.ok is true
   - On !response.ok: log warning with status code, INSERT into sync_failures
   - On fetch throw (existing catch): also INSERT into sync_failures
